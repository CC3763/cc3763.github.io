**摘要：** 本文探讨如何在使用 AI Agent 作为个人助手时平衡个性化体验与隐私保护，以及可行的隐私策略。

## 1 背景

随着 AI Agent 越来越深入地参与我们的日常工作，一个核心矛盾逐渐显现：Agent 知道得越多越能提供个性化服务，但信息越多也意味着隐私风险越大。如何在两者之间取得平衡，是每个 AI Agent 使用者都需要思考的问题。

## 2 核心风险分析

### 2.1 数据流向

用户与 AI Agent 的交互数据通常会经过三个环节：

1. **本地存储** — 用户的偏好设置、记忆信息（如 HERMES 的 `USER.md` / `MEMORY.md`）
2. **模型提供商** — 每次对话的消息内容会被发送到云端模型进行推理
3. **第三方平台** — 通过飞书、微信等即时通讯工具传递的消息也会经过对应平台

### 2.2 主要风险

- **云端模型可见性**：模型提供商理论上可以看到所有对话内容
- **服务器入侵**：AI Agent 运行所在的服务器可能成为攻击目标
- **数据留存**：会话记录如果不定期清理，会持续累积敏感信息

## 3 应对策略

### 3.1 数据最小化原则

在 Agent 的记忆中只保存**必要的最小信息**，避免存储完整的个人身份信息：

| 建议存储 ✅ | 避免存储 ❌ |
|-----------|-----------|
| 工作习惯偏好 | 身份证号、家庭住址 |
| 常用工具/语言 | 银行账户、登录密码 |
| 时区、常用场景 | 完整的个人履历 |

### 3.2 本地模型方案（最高隐私）

对于隐私要求极高的场景，可以使用本地大模型（如 Ollama + Qwen2.5），所有数据不出本地：

```yaml
model:
  provider: ollama
  model: qwen2.5:7b
  base_url: http://localhost:11434/v1
```

### 3.3 会话数据管理

配置自动清理策略，避免数据长期累积：

```yaml
sessions:
  auto_prune: true       # 自动清理旧会话
  retention_days: 30     # 保留最近30天
```

## 4 总结

使用 AI Agent 作为个人助手时，隐私保护的核心是**数据最小化**与**合理选择模型部署方式**。对于日常使用，云端模型配合最小化信息存储已经足够；对于高度敏感的场景，切换到本地模型则可实现完全的数据自控。
